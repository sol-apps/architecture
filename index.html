How solhann.net works

One small server runs a whole family of web apps — each at its own address, each sealed off from the others. Here’s the path a change takes to get live, and the walls that keep every app in its lane.

01THE SETUP

One server, many front doors

Every app here has its own address — quiz.solhann.net, expenses.solhann.net — and behaves as though it owns the machine. In reality they all share a single rented server. The point is that none of them can tell, and none of them can reach each other.

Adding one is a single command. Getting a change live is a single push. Everything between — the checking, the delivery, the publishing — happens on its own. The rest of this page is what “on its own” actually means.

02SHIPPING

From save to live in about thirty seconds

You change something and send it once. From there it’s automatic: the change is checked for mistakes, delivered to the server, and published to the web — no manual steps, no logging in anywhere.

YOU CHANGE ITon your own computer IT’S CHECKEDmistakes caught before it ships IT’S DELIVEREDto the shared server IT’S LIVEfor everyone, in seconds

The delivery route is deliberately powerless: it can drop new files at exactly one spot and do nothing else — no reading data, no opening other apps, no running commands. If the automation were ever hijacked, there’d be almost nothing worth hijacking.

03ISOLATION

Every app in a locked room

One server, but each app runs as its own user in its own space with its own key. An app can only reach its own files. If one breaks — a bug, a runaway process, even hostile code — it stays boxed in: it can’t read the app next door, and it can’t take the machine down with it.

ITS OWN KEY

No shared master key. An app literally cannot open another app’s data — the operating system refuses at the door.

ITS OWN LIMITS

Memory and processing are capped per app. A greedy or broken one only ever slows itself down, never its neighbours.

04STORAGE

Two doors on the filing cabinet

Some apps remember things — your receipts, your saved game. Each of those gets a private database with two separate entrances, and keeping them apart is the entire point.

FRONT DOOR · VISITORS

Where the app’s users come in. Rules run on every request, so each person sees only their own records — never anyone else’s.

BACK DOOR · THE MAKER

Where you send the code that runs inside the app. It travels a separate channel that can write instructions but can’t read the data.

05THE GALLERY

It assembles itself

Publish a new app and it appears on the front page on its own — no list to maintain, no card to lay out by hand. The gallery notices the new neighbour and adds it.

This page is one of those neighbours — built, checked, and shipped in exactly the way everything above describes.
