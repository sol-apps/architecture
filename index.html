How it works

One rented server runs a handful of small web apps, each at its own address. This page is about two things: getting a change live in half a minute, and making sure no app can ever read its neighbour’s data.

01THE SETUP

One server, many front doors

Each app has its own address — quiz.solhann.net, expenses.solhann.net — and runs as if it owned the server. It doesn’t. Everything’s on one box. But an app can’t see the others, let alone reach into them.

Spinning up a new app takes one command. After that, pushing your code is the only step you do by hand. The rest runs itself.

02SHIPPING

From save to live in about thirty seconds

You edit a file and push it. That’s the last time a person is involved. A build kicks off, checks the change for anything obviously broken, copies the result onto the server, and the live site updates. Usually inside thirty seconds.

YOU CHANGE ITon your own computer IT’S CHECKEDmistakes caught before it ships IT’S DELIVEREDto the shared server IT’S LIVEfor everyone, in seconds

The thing doing the copying has almost no power on the server. Its login can write files into one folder and that’s it — no shell, nothing it can read back. If those credentials leaked tomorrow, the worst anyone could do is push a bad version of an app that already trusts them.

UNDER THE HOOD

The deploy credential is an SSH key with a forced command in authorized_keys: command="rrsync /var/www/apps". Whoever holds it can run exactly one thing — rsync, scoped to that one folder. No shell, no other commands, even with the key in hand.

Server-side code for data apps rides a second key pinned to rrsync -wo (write-only), so it can drop new logic in but can’t read anything back out. The build and copy themselves run in GitHub Actions from one shared workflow.

03ISOLATION

Every app in a locked room

The apps share a machine, but not an account. Each one runs as its own Linux user that owns a single folder. When an app asks for a file in another app’s folder, the operating system refuses — the same way your laptop stops one login from opening another’s files.

That covers ordinary bugs. It also covers an app that’s actively hostile, because the design assumes any app might run code nobody vetted.

ITS OWN KEY

No shared master key. One app reaching for another’s files gets stopped by the kernel itself, not by a policy we’re trusting to hold.

ITS OWN LIMITS

Every app has a hard cap on memory and CPU. A runaway one slows itself down and leaves the rest of the box alone.

UNDER THE HOOD

Each app is a systemd service running as a dedicated user (pb-<app>), with its data folder at mode 750 — unreadable to any other user.

The service is sandboxed: ProtectSystem=strict with ReadWritePaths= set to just that folder, an empty CapabilityBoundingSet= (no root powers at all), SystemCallFilter=@system-service, and MemoryDenyWriteExecute=yes. Resource caps are MemoryMax=256M, CPUQuota=50%, TasksMax=64.

So even code running flat-out inside an app can’t step outside its folder or starve the machine. Full list of these knobs: systemd sandboxing options →

04STORAGE

Two doors on the filing cabinet

Some apps store things — receipts, scores, whatever people save. Each of those gets its own small database, with two ways in that never cross.

FRONT DOOR · VISITORS

Where visitors land. Every read runs through a rule that checks who’s asking, so people only ever get their own rows back.

BACK DOOR · THE MAKER

Where you push the app’s server-side code. That channel can add new logic but can’t read the stored data, so a bug you ship can’t quietly leak the database.

UNDER THE HOOD

Storage is PocketBase, one instance per app, bound to 127.0.0.1 behind Caddy. Front-door access is governed by rule expressions checked on every query — for example owner = @request.headers.x_owner — so an app can’t forget to scope a read.

The admin console is restricted to an allowlisted IP. The back door, where hook code lands, is the write-only key from section 02 — it can add behaviour but never read what’s stored.
